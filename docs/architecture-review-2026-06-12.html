Problem — 탐험 워프 세션(블랙아웃 320ms·busy 잠금·back 스택·exclude 윈도·키보드·하단 nav·에러 toast)이 두 호스트에 통째로 중복되고, 두 번째 호스트는 테스트된 history module을 우회해 cap 상수(10/20)까지 인라인으로 다시 박았다 — locality 없음.

  • locality: ISSUE-001류 수정이 한 곳
  • leverage: 한 interface, 호스트 N개
  • fake adapter로 세션 자체 테스트 가능
  • 키 드리프트 해소(Backspace vs ←)
  • 중복 ~150줄 삭제
  • Overlay·btnStyle 사본 2벌 → 1벌
ADR-0003(지속 Canvas·registry clear·exclude RPC)와 충돌 없음 — 오히려 락인 사항을 두 곳 대신 한 곳에서 집행하게 된다.

2. PublicScene의 seam 누수 — boolean 프롭 누적

Worth exploring in-process
app/scene/PublicScene.tsx (149줄 · 테스트 없음)
호스트 3곳: app/s/[short_code]/page.tsx · app/discover/page.tsx · GroupDiscover.tsx

Before — interface가 호스트 사정을 인코딩 (shallow)

tree · warp · keyboardFocus · bottomNav
프롭마다 "어느 호스트인가"를 알아야 함
Canvas + 카메라 + System
focused Body 라벨 (호스트 크롬인데 뷰어 안)
focus-key 배선 (테스트 없음)

실증: 라벨이 호스트 버튼에 가린 버그(7b13c94) — 수정도 boolean 프롭 추가로, 누수가 한 칸 더 깊어짐

After — 뷰어는 Universe 렌더만 (deep)

tree (+ Focus 상태 노출)
Canvas + 카메라 + System + 워프 카메라
focus 가드·registry clear — 내부로 흡수
라벨·키보드·nav 크롬 → 호스트(후보 1의 워프 세션) 한 곳에서 — 서로를 아는 채로 배치

Problem — PublicScene의 interface가 호스트당 boolean 프롭을 하나씩 늘리는 중이고(warp → keyboardFocus → bottomNav), focused Body 라벨이라는 호스트 크롬이 뷰어 implementation 안에 있어 seam 양쪽이 서로의 레이아웃을 모른다 — 방금 고친 라벨 가림 버그가 그 직접 증거.

Solution — 뷰어는 Universe 렌더와 Focus 상태 노출까지만 책임지고, 크롬(라벨·키보드·하단 nav)은 호스트 쪽 한 곳으로 모은다.

  • interface 축소: 프롭 4 → 1~2
  • locality: 크롬 충돌이 한 파일 안
  • 같은 버그 클래스 재발 차단
  • /s/[code] 경로 더 단순해짐
후보 1과 자연스럽게 한 묶음 — 워프 세션 module이 생기면 크롬의 귀속처가 저절로 정해진다.

3. SphereSync — 동기화 정책이 useEffect 클로저에 갇힘

Worth exploring local-substitutable
app/me/SphereSync.tsx (106줄 · 테스트 0)

Before — 정책 6가지가 한 클로저에

<SphereSync userId /> — 아무것도 말하지 않는 interface
· 에코 억제 (serverTree 비교)
· 1.5s 디바운스 + pending 관리
· 로드 윈도 동안 server-wins
· 첫 로그인 → 로컬을 시드로 push
· 언마운트 시 flush (유실 방지)
· 하이드레이트 후 undo 스택 clear
전부 클로저 변수 — interface 밖에서 검증 불가

After — 프레임워크 없는 sync 세션 module

sync 세션 (순수 TS) — deep
에코 억제·디바운스·server-wins·시드·flush 전부 내부
transport는 seam 너머 adapter (supabase / fake)
SphereSync 컴포넌트 = 10줄 adapter (mount/unmount → start/stop)

fake transport + fake timer로 6개 정책 시나리오를 vitest에서 직접 검증

Problem — local-first 동기화의 실질 정책(에코 억제·디바운스·server-wins·시드·flush·undo clear)이 전부 한 useEffect 클로저 안이라 interface가 아무것도 드러내지 않고 테스트가 0 — QA ISSUE-001과 같은 패턴(순수부 countNodes만 테스트, 배선은 비테스트)이 앱에서 가장 미묘한 module에 그대로 남아 있다.

Solution — 정책을 프레임워크 없는 sync 세션 module로 추출하고 transport(supabase)는 adapter로 주입, 컴포넌트는 얇은 adapter로 남긴다.

  • interface가 정책을 드러냄
  • 테스트가 interface를 통과
  • 유실 시나리오(편집 후 이탈) 검증 가능
  • 멀티탭(D7 연기분) 착수 지점이 한 곳
eng-review D3는 "local-first + 디바운스"만 락인 — 구현 형태는 자유. 충돌 없음.

Speculative in-process
useSphereStore · createPublicSphereStore · SphereSync · OrbitalBody 타입 · "tree" 변수 전반
CONTEXT.md: «"sphere"는 개별 노드를 가리킬 때도 쓰지 않는다» — 이미 락인된 금칙어

Before — 한 개념, 세 이름

spheretreeuniverse→ Universe
OrbitalBodynodebody→ Body

개념 하나를 추적하려면 grep을 세 번 — AI 탐색성·신규 합류 비용 모두에 마찰

After — 용어집이 곧 코드 인덱스

Universe — 스토어·타입·프롭 전부
Body — 노드 타입과 변수

Problem — CONTEXT.md가 금지한 "sphere"가 코드 표면 전체(스토어·타입·컴포넌트 이름)에 남아 있어 도메인 용어집과 코드가 서로 다른 언어를 쓴다.

Solution — 코드 레벨 이름을 Universe/Body로 수렴한다(기계적 rename, DB는 제외).

  • 한 개념 = grep 한 번
  • 용어집과 코드 일치
rename 전용 PR은 열린 브랜치와 충돌 비용이 큼 — 후보 1/2를 건드릴 때 그 파일들만 함께 개명하는 점진 방식 권장.

Top recommendation

후보 1 — 워프 세션 deep module

증거가 가장 신선하다: 직전 두 버그(ISSUE-001 back 스택, 라벨 가림 7b13c94)가 모두 이 seam에서 났고, 후자는 두 호스트에 똑같이 적용해야 했다. adapter가 이미 둘(공개 풀·그룹 풀)이라 seam이 real이고, 후보 2(크롬 귀속)와 후보 4(adapter 승격)가 공짜로 딸려온다. 세 번째 탐험 모드(예: M6에서 테마별·팔로우 탐험)가 생기는 순간 중복은 세 채가 된다 — 지금이 가장 싸다.
